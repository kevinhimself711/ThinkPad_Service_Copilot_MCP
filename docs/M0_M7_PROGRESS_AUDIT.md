## Detailed Findings

### M0 Repository Bootstrap

M0 achieved the required upstream relationship instead of creating an unrelated repository. The local project keeps the upstream baseline and then layers ThinkPad-specific work on `thinkpad-hmm-domain`.

Evidence:

- `docs/M0_BASELINE.md` records upstream URL and baseline commit.
- `origin/main` remains at the bootstrap commit.
- Current development branch is `thinkpad-hmm-domain`.

No M0 gaps remain.

### M1 HMM Risk Spike

M1 completed the intended risk validation for official Lenovo HMM sources. The local data flow uses official Lenovo discovery rather than mirrors:

```text
Lenovo self-repair page
-> Guid / ParentGuids
-> api/v4/contents/recommendmanual
-> hardwareMaintenanceManual.pdfs
```

M1 proved the risks are real and worth engineering around:

- table candidates exist at scale;
- figure and raster fallback behavior matters;
- model/generation metadata must be explicit;
- FRU procedure sections and prerequisites need dedicated handling.

Carry-forward risk:

- M1 did not manually certify row/column quality or every diagram/procedure. That was intentional spike scope, not a failure.

Verdict: `complete_with_risk`.
