# ThinkPad Service Copilot MCP · 可行性评估

> 评估对象:基于母项目 `jerry-ai-dev/MODULAR-RAG-MCP-SERVER`,用「ThinkPad 硬件维修手册(HMM)知识助手」场景打造自己的 RAG 项目。
> 语料:ThinkPad **T 系 + X1 + E 系 + P 系,2018–2025**,约 50 本 HMM PDF。

---

## 0. 结论先行

- **总体:高度可行,且是少见的"顺毛"匹配。** 母项目最难造假的五块能力——PDF 原生摄取、多模态 image-to-text、混合检索、按权威性/版本精排、引用溯源——这个场景**每一块都用满**,不存在"绕开项目核心能力"的浪费(这正是上一个 DevDocs 方案被否的原因)。
- **有 4 个真实技术风险,全部集中在摄取环节**(表格解析 / 图纸抽取与 caption 价值 / 版本型号元数据 / 拆卸顺序被切断)。它们**不是拦路虎,而是面经考点的活体**——你解决它们的过程就是面试答案。
- **纪律要求:先做"风险验证 spike"**。用 5–8 本手册先把这 4 点验证一遍(约 2–3 天),再决定全量铺 50 本。别闷头做到最后才发现图纸抽不出来。
- **反 toy 护城河:可量化对比实验**——接 MCP vs 不接,coding agent 在 ThinkPad 维修任务上的版本正确率/Faithfulness 差多少。把"有没有用"变成一个数字。
- **判定:Go。**

## 1. 逐模块契合度(对照母项目 DEV_SPEC)

| 母项目模块 | 场景怎么用 | 契合度 | 备注 / 要做的 |
|---|---|---|---|
| Loader(MarkItDown PDF→MD + PyMuPDF 抽图) | HMM 就是 PDF,原生走这条 | ✅ 原生 | 风险见 R1/R2 |
| Splitter(RecursiveCharacterTextSplitter) | 按 FRU 编号段(1010/1110…)切 | ⚠️ 需调 | HMM 有强结构 + 拆卸依赖,naive 切会切断 → R4 |
| Transform·ChunkRefiner | 清页眉页脚/版次水印/OCR 噪声 | ✅ | 老机型扫描件用得上 |
| Transform·MetadataEnricher | 加 `model`/`gen`/`version`/`fru_id`/`section`/`error_codes` | ✅✅ 加场景字段 | 型号是**多值**字段 → R3 |
| Transform·ImageCaptioner | 给拆装图/爆炸图配文 | ⚠️ 价值需校准 | 价值是"取回该图返回",精确规格在文本 → R2 |
| Embedding 双路(Dense + BM25) | FRU 号/错误码 `0271`/扭矩 `0.294Nm` → BM25;"换板后充不进电" → Dense | ✅✅ 教科书级互补 | 场景天然适配混合检索 |
| Storage(Chroma + BM25 倒排 + ImageStorage) | 原图存本地,命中后返回 | ✅ | — |
| 检索 + RRF 融合 | Dense/Sparse 并行融合 | ✅ | — |
| Rerank(None/Cross-Encoder/LLM,可插拔) | 按**型号/代/版次权威性**精排 | ✅✅ 要写规则 | 面经第4题"为什么规则精排"的活体答案 |
| MCP Server(stdio + tools + citation + 图 Base64) | 技师在 IDE/桌面问,答案带页码带图 | ✅✅ | tools 改名 `query_thinkpad_service` / `get_fru_procedure` / `list_models` |
| Dashboard / Trace(六页面) | 现成,直接可观测 | ✅ | 低风险 |
| Eval(Ragas / Custom,可插拔) | 手册即 ground truth | ✅✅ | Golden Test Set 好造 |

**结论**:除了 Splitter 和 Rerank 需要为场景写定制逻辑、ImageCaptioner 的价值需要校准,其余模块都是**现成直接用**。没有任何一块是"为了用而硬凑"。

## 2. 数据可行性

- **来源(已验证可下)**:Lenovo 官网 `pcsupport.lenovo.com`(每个型号都发 HMM PDF,权威源)+ ManualsLib + 镜像站。
- **数量与边界**:约 50 本 HMM;一本常覆盖多型号(如 T530/T530i/W530 一本、E14/R14/E15/S3 Gen2 一本、P1 Gen4/X1 Extreme Gen4 一本)→ 覆盖几十个型号。
- **规模估算(诚实版)**:50 本 × ~120 页 ≈ 6000 页。按 chunk_size≈1000 字符(英文),粗估 **1–2 万 chunk**(比面经里的 5 万低,因为手册图表多、纯文本密度不如散文;**以你实测为准**)。向量库:1.5 万 × 1536 维 × 4B ≈ 90MB dense + Chroma 存原文 → 几百 MB;BM25 倒排 JSON 几十 MB。**ChromaDB 完全扛得住**,对齐面经的规模/选型答案。
- **多值型号 metadata**:一本管多型号是真实摄取细节(metadata 的 `models` 要存数组),正好是面经"元数据有什么"的料 → R3。
- **难点天然存在**:新机型干净数字 PDF、老机型偏扫描质量差(喂去噪+质量预检);跨代近重复内容(喂版本精排+去重)。不用刻意造,语料本身就带。
- ⚖️ **合规**:HMM 是 Lenovo 版权,个人/学习用途本地 RAG 没问题;公开 demo 时引用并链接 Lenovo 原文、不整本转载。

## 3. 四个真实技术风险 + 缓解(评估的核心)

> 这部分最显工程成熟度。每个风险都在 Phase 1 的 spike 里先验证。

### R1 · 表格解析(中风险)
**问题**:错误码表(`0271 → FRU/action`)、FRU 表、螺丝规格表(数量/颜色/扭矩)是**核心问答来源**,但 PDF 表格经 MarkItDown 容易解析成乱序文本,行列错位 → 检索到也答不准。
**缓解**:对表格区域用 `pdfplumber`/PyMuPDF 的表格抽取单独处理,把每行结构化后再注入 chunk(或存进 metadata 的 `error_codes`/`screw_specs`);Phase 1 先抽 2–3 张典型表验证。
**面经价值**:正是"PDF 多排版怎么处理 / 数据质量差怎么办"。

### R2 · 图纸抽取与 caption 价值(中高风险,最该早验)
**问题**:① 拆装图可能是**矢量线稿**,PyMuPDF 的嵌入图抽取(针对栅格 XObject)可能**漏抽或抽出空白** → 多模态没素材;② 单张线稿独立 caption 信息有限(精确螺丝数/扭矩在文本不在图)。
**缓解**:① 对图密集页做**整页/图区域栅格化兜底**(rasterize),保证拿得到图;② 把图**前后文本注入 Vision prompt**(母项目已有此设计),让 caption 带上下文;③ **明确多模态的价值定位**——是"按'X 的拆卸图'取回正确图纸返回给技师",不是从图里读精确规格。Phase 1 抽 ~10 张图验证 caption 质量再决定 prompt。
**面经价值**:"图片处理怎么设计、为什么 image-to-text 不用 CLIP"。

### R3 · 版本/型号元数据(中风险,是精排的命门)
**问题**:规则精排和型号消歧全靠准确的 `model`/`gen`/`version` 标注,而一本 HMM 覆盖多型号、标题页格式不一(如"T530/T530i/W530,第三版,2013-03")。标错 → 精排把错代的手册顶上来。
**缓解**:写一个从**标题页 + 文件名**抽取「覆盖型号列表 + 版次/日期」的小流程,结果人工抽检;存成 `models:[...]`、`gen`、`edition`、`year`。
**面经价值**:"元数据都有什么 / 增强部分做什么"。

### R4 · 拆卸顺序依赖被切断(中风险,也是个加分机会)
**问题**:FRU 拆卸有**前置依赖**("拆 1110 键盘 bezel 前,先按序拆 1010 电池、1040 M.2…"),naive 定长切会把"依赖清单"和"拆卸步骤"切散,检索回来缺前置 → 技师照做会出错。
**缓解**:**结构感知切分**(以 FRU 编号段为切分单位)+ chunk metadata 存 `fru_id` 与 `prerequisites`;或检索后用 `chunk_index` 把前置步骤拼回。
**机会**:HMM 的 FRU 编号 + 前置清单结构非常规整,你可以顺手建一张**前置依赖图**,让答案不只给单步、还给完整拆卸链——这是个能讲的差异化深度。
**面经价值**:"分块怎么做、完整流程被切断怎么办"。

> 这 4 个风险**一一对应面经**的 PDF 解析 / 分块 / 元数据 / 混合检索题。换句话说:风险清单 = 面试题清单。

## 4. 评估与可观测可行性

**可评估(手册即真值,好造)**——Golden Test Set 题型(全可核对):
- 精确型:"E15 Gen2 LCD 铰链螺丝扭矩"(0.294Nm…)、"E14 Gen2 错误码 0271 含义"。
- 流程型:"T530 拆键盘 bezel 前要先拆哪些 FRU"(查前置清单)。
- 多模态型:"P1 Gen4 散热风扇的拆卸图"(看 caption 是否被检索到 + 图是否返回)。
- **版本陷阱型**:"X1 Carbon 拆电池"(语料含 Gen7/Gen9)→ 看是否返回匹配代。
- 指标:Ragas Faithfulness / Answer Relevancy / Context Precision + Hit Rate@K。

**反 toy 的关键——可量化对比实验**:
让 Codex/Claude Code 跑一批 ThinkPad 维修编码/问答任务,**接你的 MCP vs 不接**,对比:① 引用的 FRU 号/扭矩/拆卸顺序的**型号正确率**;② 答案 Faithfulness;③ 幻觉率(瞎编不存在的 FRU/步骤)。一个明确的提升数字,直接把"有没有用"坐实——这是这个项目最有力的反 toy 动作,也对上你要的"可评估"。

**可观测(现成 Dashboard 三屏演示)**:
- `0271` 错误码 query → Sparse(BM25)命中错误码表、Dense 漏 → 演示"为什么混合检索"。
- `X1 Carbon 拆电池` query → rerank 前后排名,匹配代手册被顶上来 → 演示"为什么规则精排"。
- 各阶段耗时瀑布图 → 回答"每个环节耗时多少"。

## 5. 工作量与时间

两条路径(详见《详细施工规划》M0–M9):
- **Path A(推荐,深度)**:`clean-start` 分支,改 DEV_SPEC → Skill 驱动 AI 按 Spec 建 → 逐任务人工 Review。最扛面试。
- **Path B(快速)**:`main` 分支,跑通 setup → 只改场景层(Loader 增强/分块/精排规则/metadata/prompt)→ 重跑。

**场景特有的额外工作量(在母项目之上)**:
- R1–R4 的定制(表格抽取、图栅格化兜底、型号元数据抽取、结构感知切分)。
- 语料采集 + 多值型号打标(~50 本)。
- 规则精排(型号/代/版次权威性)。
- Golden Test Set(≥30 条)+ 量化对比实验。

**时间预算(诚实)**:
- 业余(每天 2–3h):Path A 约 **7–11 周**到可评估;+深度扩展再 2–4 周。比通用场景多约 1–2 周(就是 R1–R4 的活)。
- 全职冲刺:约 **2.5–3.5 周**到可评估。
- **第一周必做**:Phase 0/1 的**风险验证 spike**(5–8 本手册,验 R1–R4),再决定全量。

## 6. 面试可行性(这个项目能正面回答哪些面经狠题)

| 面经狠题 | 这个项目的真实答案 |
|---|---|
| PDF 多排版/数据质量差怎么办 | R1 表格抽取 + 质量预检 + ChunkRefiner;扫描件 OCR |
| 分块怎么做/流程被切断怎么办 | R4 结构感知切分 + 前置依赖 + chunk_index 拼回 |
| 多模态怎么设计/为什么不用 CLIP | R2 image-to-text 取回图纸,自然语言搜图;CLIP 不懂线稿逻辑 |
| 为什么混合检索/举例 | FRU 号/`0271`/扭矩走 BM25,症状走 Dense(真实例子张口就来) |
| 为什么规则精排不用 cross-encoder | 按型号/代/版次权威性加权,修错有报废风险,可解释性优先 |
| 元数据有什么/增强做什么 | model(多值)/gen/version/fru_id/error_codes/screw_specs |
| 为什么自建不用 ES/LlamaIndex | 本地化、几万 chunk、深度可控 |
| 规模/向量库多大/扩展 | ~1–2 万 chunk、几百 MB、Chroma;扩展 Qdrant/Milvus |
| 怎么评估/减少幻觉 | 手册真值 Golden Set + Ragas + 量化对比(接/不接 MCP) |
| 这数据适合 RAG 吗 | 事实强、私有(手册集)、需溯源、太细 LLM 没背 → 四条全中 |

## 7. 最终结论 + 启动建议

**Go。** 这个场景对母项目是顺毛匹配,风险可控且恰好都是面经考点,数据可得、可评估、可观测,反 toy 有量化抓手。

**第一步(本周)**:做风险验证 spike——
1. 下 5–8 本代表性 HMM(覆盖新/老、含表格、含图、含跨代):比如 T14 Gen2、T14 Gen3、X1 Carbon Gen9、E15 Gen2、P1 Gen4、T470s。
2. 手动验证 R1–R4:表格抽得对吗?图抽得出+caption 有用吗?标题页型号抽得准吗?FRU 段切得整吗?
3. 验证通过 → 按《施工规划》进 M1(改 DEV_SPEC)→ 全量铺 50 本。

> 一句话:不是"能不能做",而是"先验证 4 个摄取风险,然后照施工规划做"。这 4 个风险解决了,你这个项目就同时具备了**真实用途**和**面经里被反复追问的全部深度**。
